# Chapter 10 — Data Truth States

*Volume 5: Data & Sensors | OVERSCITE Global User Manual*

---

## 10.1 The Problem Truth States Solve

Most evolving software platforms have a visibility problem: features are partially built, data sources are partially connected, and interfaces exist at various stages of maturity. Users cannot tell the difference between a dashboard widget that displays real live data and one that displays hardcoded sample data. Both look identical. Both use the same design language. The operator has no way to know whether they are making decisions based on real information or polished placeholders.

This ambiguity is dangerous in professional domains. An inspector who trusts a "compliance score" because it appears in a professional interface — without knowing that the score is computed from mock data — may make real-world decisions based on fictional inputs. This is exactly the kind of failure that OVERSCITE's truth-state system is designed to prevent.

---

## 10.2 The Five Truth States

Every data-bearing surface in OVERSCITE carries a **truth-state declaration** — a label that communicates the maturity, reliability, and provenance of the data it displays. There are five possible truth states:

### [LIVE]

**Definition**: The data is sourced from a real, active data provider. The interface logic is fully operational. Interactions produce real, consequential results. The display accurately reflects current system state.

**What it means for you**: You can rely on this data for professional decisions. It is as real as the system can make it. The Weather Command Center's current conditions are [LIVE] because they come from a real weather API. The Dashboard's greeting is [LIVE] because it uses your actual login identity and the current time.

**Technical basis**: The data source is a production API, a connected database collection, or a real-time service. No synthetic generation, no mock objects, no static seed data.

### [PARTIAL]

**Definition**: The interface logic and data source are partially connected. Some aspects of the feature work with real data while others remain scaffolded or pending integration.

**What it means for you**: You can use the feature, but expect that some aspects may not behave fully. The parts that work are reliable. The parts that don't are clearly indicated. For example, the Locations OverSCITE system is [PARTIAL] — its architecture is complete, but full map rendering requires API key configuration.

**Technical basis**: The component has real integration code but depends on external configuration, API keys, or backend features that are not yet universally available.

### [MOCK]

**Definition**: The data is synthetic — generated for structural simulation and UI validation. The display looks real but the underlying data did not come from actual operations.

**What it means for you**: Do not use this data for professional decisions. It exists to demonstrate the interface structure and to support development of downstream features. The data is realistic in format but fictional in content.

**Technical basis**: The data source is a mock service, a seed data file, or a static array of sample records. The logic that *processes* the data may be real, but the data itself is not.

### [CANDIDATE]

**Definition**: The data or output has been generated by the system (typically by a LARI engine) but has not yet been reviewed and approved by a human operator. It is a proposal, not a record.

**What it means for you**: Treat this as a draft. Review it carefully. AI-generated findings, report drafts, and risk assessments start as [CANDIDATE] and become [ACCEPTED] only when you explicitly approve them. The transition from CANDIDATE to ACCEPTED is an audited event.

**Technical basis**: The data was produced by an engine handler and returned through the governed pipeline, but the human approval gate has not yet been passed.

### [ACCEPTED]

**Definition**: The data was reviewed and approved by a human operator. It has been locked, signed, or otherwise committed as an authoritative record.

**What it means for you**: This data has human authorization behind it. It has passed through an approval gate, and that approval event is recorded in the audit trail. Inspection reports that have been finalized are [ACCEPTED]. Findings that the inspector has confirmed are [ACCEPTED].

**Technical basis**: The human authorization event is recorded in a BANE SDR. The data may be locked against further modification depending on the domain rules. The transition itself is timestamped and attributed.

The following matrix shows the truth-state classification of every authenticated route in OVERSCITE, as documented in the Implementation Truth Matrix:

| Route | Truth State | Data Source | Notes |
|-------|-------------|-------------|-------|
| Dashboard | [LIVE] | UI Tokens, static | Truthful greeting, standardized transparency |
| Library | [MOCK → FUNCTIONAL] | `library-service.ts` | Real document records, functional simulated upload |
| Calendar | [MOCK → FUNCTIONAL] | `calendar-service.ts` | Persisted booking records, data-driven grid |
| Finances | [MOCK → FUNCTIONAL] | `finance-service.ts` | Billing records visibility, truthful sub status |
| Messaging | [PARTIAL] | Pending service | Moving from loader to thread list |
| Admin | [MOCK] | Placeholder | Static health placeholders, non-functional toggles |
| Marketplace | [PARTIAL] | Static metadata | Capability registry concept, "Request Access" logic |
| Inspections | [MOCK → FUNCTIONAL] | `mockData.ts` | Table data visible, detail view functional |

## 10.4 How Truth States Appear in the Interface

Truth states manifest in the interface through several mechanisms:

**Page Identity Band badges**: Each route's Page Identity Band includes a truth-state badge. This badge uses the platform's monospace typography and a neutral/muted visual treatment — it is informational, not promotional.

**Data panel markers**: Individual data panels (such as the Live Radar tab in Weather) may carry their own truth-state indicators when their maturity differs from the page-level classification. A [LIVE] page may contain an individual component that is [PARTIAL].

**Module header descriptions**: The four-sentence mission descriptions in module headers include language that honestly describes the feature's scope and readiness.

**Inline disclaimers**: Where mock data is present, the system may include an inline text note (such as "Sensor calibration logs and force-fix events will appear here") that indicates the data is structural rather than operational.

## 10.5 The Truth-State Transition Lifecycle

Truth states are not static. They evolve as features mature:

```
[MOCK] → [PARTIAL] → [LIVE]
                         ↕
              [CANDIDATE] → [ACCEPTED]
```

**Development transitions**: A feature begins as [MOCK] during initial development, transitions to [PARTIAL] when integration begins, and reaches [LIVE] when its data source is fully connected and validated.

**Operational transitions**: Within a [LIVE] feature, individual data records may start as [CANDIDATE] (system-generated proposals) and transition to [ACCEPTED] (human-approved records). This transition is the operational expression of the Human Authority Doctrine.

**Key principle**: A feature's truth state can only advance forward in response to real implementation progress. It is never inflated. A feature does not claim [LIVE] status until it genuinely earns it through connected data sources and validated logic.

---

## 10.6 Why This Matters

**For professional trust**: When you see [LIVE] on the Weather Command Center, you know the temperature reading is real. When you see [MOCK] on the Admin panel, you know the toggle switches do not do anything yet. This clarity prevents the most insidious kind of software error — the one where the operator trusts data that is not trustworthy because the interface gave no indication of its unreliability.

**For legal protection**: In the event of a dispute or audit, the truth-state system demonstrates that the platform never misrepresented its capabilities. If a finding was marked [CANDIDATE], the audit trail shows it was a system-generated proposal, not a human-attested observation. If it was marked [ACCEPTED], the trail shows who approved it and when.

**For system integrity**: Truth-state enforcement prevents the development team from accidentally shipping a feature that looks production-ready but is not. It is a governance discipline applied to the development process itself, not just to operator-facing data.

---

## 10.7 Chapter Summary

OVERSCITE's truth-state system assigns one of five maturity labels — [LIVE], [PARTIAL], [MOCK], [CANDIDATE], [ACCEPTED] — to every data-bearing surface in the platform. These labels are visible in the interface and auditable in the system records. They ensure that operators always know the maturity and reliability of the information they are working with.

This transparency is a fundamental governance feature. It eliminates the ambiguity that plagues evolving platforms and protects operators from making real-world decisions based on fictional data.
