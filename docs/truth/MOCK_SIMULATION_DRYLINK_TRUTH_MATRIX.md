# Mock / Simulation / Dry-Link Truth Matrix

## Purpose
This document defines the distinction between Mock, Simulation, and Dry-Link truth-states to prevent architectural drift and ensure transparent testing.

## Truth-State Definitions

### MOCK
- **Definition**: Hardcoded or randomized data with no underlying logic or sensor binding.
- **Usage**: Early UI prototyping, unit testing of isolated components.
- **Reliability**: None. Only for visual/structural validation.

### SIMULATION [v0.1.2 Baseline]
- **Definition**: Synthetic data generated by the ScingularRuntimeEngine based on high-fidelity models and physics envelopes.
- **Usage**: Autonomy validation, mission planning, operator training.
- **Reliability**: High within model boundaries. No external dependencies.

### DRY-LINK [v0.1.2 Optional]
- **Definition**: Metadata-only synchronization with physical TelePort hardware. Read-only telemetry intake.
- **Usage**: Site-specific readiness check, connectivity validation.
- **Reliability**: Real-world telemetry truth; Zero command authority.

### LIVE / ARMED [LOCKED]
- **Definition**: Two-way C2 connectivity with physical hardware execution enabled.
- **Usage**: Prohibited in current posture.
- **Reliability**: Total real-world truth and accountability.

## Truth-State Promotion Rules
1. **Mock → Simulation**: Requires implementation of model logic and engine binding.
2. **Simulation → Dry-Link**: Requires authorized hardware synchronization.
3. **Dry-Link → Live**: Requires Director authorization and formal BANE phase gate approval.

## Enforcement
The `OverHUD` and `ArcHive™` records must explicitly label the active truth-state. Any attempt to pass Mock data as Simulation, or Simulation as Live, is a governance violation.
